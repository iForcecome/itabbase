# ITabBase Admin 后台完整规划（v2 - 动态集合版）

## 定位

借鉴 NocoBase 的 **collection-driven + 混合模式**后台管理思想。系统集合代码定义，业务集合动态创建（API/UI/AI Skill），用户项目零代码变更即可新增表和 CRUD。明确不做：拖拽式 UI 编辑器、schema-driven 运行时 UI 渲染、前台页面配置能力（见 [docs/itab-base方案.md](docs/itab-base方案.md) 13）。

## 架构总览

```
用户创建业务集合的 3 种方式：
  1. AI Vibe Coding → 通过 Skill 调 POST /api/meta/collections
  2. Admin 后台 UI  → 集合管理页面可视化配置
  3. Go 代码(可选)  → RegisterCollection() 适用于系统级扩展

                  ┌──────────────────────────────────┐
                  │         itabbase kernel           │
                  ├──────────────────────────────────┤
                  │  内置集合 (Go 代码定义, 不可删除)   │
                  │  users, roles, user_roles,        │
                  │  system_settings,                 │
                  │  _collections, _fields ← 新增     │
                  ├──────────────────────────────────┤
                  │  动态集合 (DB 存储, API 管理)       │
                  │  启动时从 _collections 加载        │
                  │  API 创建时自动 CREATE TABLE       │
                  │  修改字段时自动 ALTER TABLE        │
                  ├──────────────────────────────────┤
                  │  统一 CRUD + ACL + Hooks          │
                  └──────────────────────────────────┘
```

NocoBase 的混合模式参考（来源：[NocoBase plugin-data-source-main](file:///Users/king/localai/source/packages/plugins/@nocobase/plugin-data-source-main/src/server/server.ts)）：
- 代码定义的集合启动时先加载到内存
- DB 定义的集合在 `beforeStart` 时从 `collections`/`fields` 表加载并合并
- 同名集合：DB 定义 merge 进已有的内存集合（`updateOptions`）
- 代码集合可标记 `uiManageable` 同步到 DB 让 UI 管理
- 创建集合时 `afterSave` 自动跑 `collection.sync()` 建表

## P0：动态集合核心（最高优先级）

这是整个平台的基石，必须先做。

### Kernel 侧改动 ([server/](server/))

**1. 新增 `_collections` + `_fields` 系统表**

```go
// 内置集合，和 users/roles 一样在 registerBuiltins() 中注册
var metaCollections = Collection{
    Name:    "_collections",
    Display: "集合定义",
    Fields: []Field{
        {Name: "name",    Type: TString, Required: true},  // 表名，唯一
        {Name: "display", Type: TString},                   // 显示名
        {Name: "icon",    Type: TString},                   // 图标
        {Name: "sort",    Type: TInt,    Default: 0},       // 排序
    },
}

var metaFields = Collection{
    Name:    "_fields",
    Display: "字段定义",
    Fields: []Field{
        {Name: "collection_name", Type: TString, Required: true}, // FK → _collections.name
        {Name: "name",            Type: TString, Required: true}, // 字段名
        {Name: "type",            Type: TString, Required: true}, // string/int/float/bool/text/datetime/belongs_to/has_many
        {Name: "display",         Type: TString},                 // 显示名
        {Name: "required",        Type: TBool,   Default: false},
        {Name: "default_value",   Type: TString},                 // JSON 编码的默认值
        {Name: "max_len",         Type: TInt},
        {Name: "target",          Type: TString},                 // 关联目标集合
        {Name: "through",         Type: TString},                 // has_many FK 列名
        {Name: "sort",            Type: TInt,    Default: 0},     // 字段排序
    },
}
```

**2. 动态集合 API**

新增 meta API endpoints（在 [server/meta.go](server/meta.go) 或新文件 `dynamic_collection.go`）：

- `POST   /api/meta/collections` — 创建集合（写 `_collections` + `_fields` 行 → 自动 CREATE TABLE → 注册 CRUD 路由）
- `PATCH  /api/meta/collections/:name` — 修改集合（display/icon/sort）
- `DELETE /api/meta/collections/:name` — 删除集合（DROP TABLE + 删除元数据行）
- `POST   /api/meta/collections/:name/fields` — 添加字段（写 `_fields` → ALTER TABLE ADD COLUMN）
- `PATCH  /api/meta/collections/:name/fields/:fieldName` — 修改字段元数据
- `DELETE /api/meta/collections/:name/fields/:fieldName` — 删除字段（需确认）

安全约束：
- 内置集合（`isBuiltin` 标记）不可通过 API 删除
- 字段删除在 dev 模式可自动执行，prod 需显式确认
- 集合名不可与内置集合或保留名（`auth`, `admin`, `meta`）冲突

**3. 启动时加载动态集合**

修改 `Kernel.Mount()` 流程：

```
1. registerBuiltins()          — 注册系统集合（含 _collections/_fields）
2. syncCollections()           — 同步系统表到 DB
3. loadDynamicCollections()    — 从 _collections + _fields 读取 → 构建 Collection → 注册到 k.collections
4. syncCollections()           — 同步动态集合的表结构
5. 挂载所有 CRUD 路由
```

**4. 现有 `/api/meta/collections` 合并输出**

当前已有的 `handleMetaCollections` 返回所有集合 meta。改造后需要：
- 同时返回代码定义的集合 + DB 定义的动态集合
- 增加 `source` 字段区分来源（`"builtin"` vs `"dynamic"`）
- 增加 `manageable` 字段标记是否可在 UI 上管理

### Admin 侧改动 ([admin/](admin/))

**5. 集合管理页面 `/collections`**

核心页面，类似 NocoBase 的 Collection Manager：

- **集合列表**：所有集合一览，区分系统/动态，显示字段数量、记录数
- **创建集合**：表单填写 name/display → 调 API 创建
- **字段管理**：点进集合 → 展示字段列表 → 可添加/编辑/删除字段
- **字段类型选择器**：string/text/int/float/bool/datetime/belongs_to/has_many
- **字段配置**：必填、默认值、最大长度、关联目标等

**6. 侧边栏改造**

```
侧边栏菜单结构：

📊 仪表盘
━━━ 数据管理 ━━━
  📁 [动态集合1]    ← 从 meta/collections 动态生成
  📁 [动态集合2]       source=dynamic 的集合
  📁 ...
━━━ 系统管理 ━━━
  🗂 集合配置        ← 管理集合和字段
  👥 用户管理
  🔐 角色权限 (P2)
  📋 操作日志 (P2)
  ⚙️ 系统设置
```

关键：侧边栏"数据管理"分组只显示动态集合（用户业务数据），系统集合（users/roles 等）通过专门的管理页面操作。

**7. CollectionView 增强**

- 表格列从 `meta.fields` 驱动，按 `sort` 排序
- 列标题显示 `field.display || field.name`
- 支持 `filter[field]=value` 筛选
- 支持 `sort=field` / `sort=-field` 排序
- `belongs_to` 字段显示关联记录名称（通过 `include`）
- `has_many` 显示计数，点击展开
- 批量删除

## 关键实现决策

- **混合模式**：参考 NocoBase，系统集合 Go 代码定义 + 业务集合 DB 存储，统一通过 `/api/meta/collections` 输出
- **动态路由注册**：kernel 启动时从 DB 加载集合定义，动态注册 CRUD 路由；运行时创建集合也需热注册路由
- **用户零代码**：创建集合完全通过 API，用户项目不需要 `RegisterCollection`（但仍支持代码注册用于系统级扩展）
- **AI Skill 对接**：提供 Cursor Skill 封装集合管理 API，让 AI 在 vibe coding 时自动调用
- **参考实现**：NocoBase `plugin-data-source-main`（混合模式）+ PocketBase（Go 单二进制）
